# GDML: GDOC MARKUP LANGUAGE

***@Summary:***  \
gdoc compiles documents tagged with the GDML (gdoc markup language) to generate structured objects.
gdoc subcommands reference these structured objects and provide user applications.
The structured object generated by this process is referred to as the Gdoc Object.
This document defines the basic structure of the Gdoc Object and the Gdoc markup language.
Additionally, it specifies the rules for parsing GDML documents in the PandocAST format.

## 1. GDOC OBJECT

***@Summary:***  \
All information extracted from a document is an object with a mutual parent-child relationship and properties.
Each object has a name, and it is also possible to reference objects in other namespaces by using a fully qualified name consisting of its own name and an ancestor's name concatenated by `.` or `::`.
Basic concepts of Gdoc Object are this parent-child relationship, name resolution by fully qualified name, and properties.
This part defines these three concepts and classes, the prerequisites for defining the gdoc markup language.

### 1.1. Gdoc Object Classes

A class diagram for explaining the concept of Gdoc Object is below.

<br>

| [![@ GdocObjectClasses](./_puml_/GdocMarkupLanguage/GdocObjectClasses.png)](./GdocMarkupLanguage.puml) |
| :-----: |
| [@fig 2.1] Gdoc Object Classes |

<br>

All objects in Gdoc Object are derived from `BaseObject` Class.
For example, each `Requirement` in requirement definition documents is a `BaseObject`, and a `Section`, an element of hierarchical document structure, is also a `BaseObject`.
And `Section` inherits to `GSN:Goal`, `GSN:Strategy`, etc.

### 1.2. Parent-Child Relationship

`BaseObject` class has a reference to one parent and zero or more children, thus forming a tree structure.

### 1.3. Id, Name, And Namespace

Gdoc Objects have a name to identify.
Usually, it's a short string of specific characters called an id.
Gdoc Objects can also have aliases using any characters, including white spaces.
These names, both id and aliases, are unique within the same namespace.

Multiple children object sharing one Gdoc Object as their parent must have different names.
That is, every object provides a namespace for its child objects.

Public/Private attributes can be assigned to each Gdoc Object.
Private Gdoc Objects cannot be referenced from outside the namespace (parent object).
It can be referenced by child objects.

The characters available for id are almost identical to those available for Python identifiers.

> [the uppercase and lowercase letters A through Z, the underscore _ and, except for the first character, the digits 0 through 9.](https://docs.python.org/3/reference/lexical_analysis.html#identifiers)

- In addition, in GDML, the first character may also be a number. Only numeric IDs are also allowed.
- We plan to allow the use of `$` as well.

### 1.4. Name Resolution And Fully Qualified Name

When referring to an object by name in a namespace, if it was not found in that namespace, it is searched for in the parent namespace.
It is repeated until the root object, which is the top-level namespace of the document.

When specifying object names, including the hierarchy of namespaces, concatenate the names from the top object to the target object with `.` or `::` to represent the hierarchy as `path.to.the.target` or `path::to::the::target`.

That long name does not necessarily have to start with the root object but must only be long enough to allow name resolution.

If you want to explicitly specify the long name from the root of the document, prefix it with `#` like `#path.to.the.target` to indicate that it is a Fully Qualified Name from the root.

### 1.5. Property

Gdoc object can have unique properties.
Properties store data in a key-value format.

Property names and child object names are managed in separate and independent namespaces, allowing the existence of child objects and properties with the same name.

Keyは、値以外にSubkeyで指定される子プロパティを持つことができ、階層化できる。

以下は理解のための概念Code.

  ```js
  // [@code 1 lang=js] Conceptual Code for understanding
  prop.key = "1"
  prop.key.subkey = "2"
  prop.key.array = "3"
  prop.key.array = "4"

  prop --> {
    "key": {
      "": "1",              // 空文字列の名前は、Key自身の値を示す。
      "subkey": "2"
      "array": ["3", "4"]   // 同じキーに複数の値をセットすると配列になる。
    }
  }
  ```

### 1.6. Gdoc Object Type and Class

Gdoc Objectは、クラスを指定することができる。
Classは、CategoryとTypeで特定される。

アプリケーションごとのニーズに応じてカテゴリは追加され、実装上はプラグインとしてgdco本体とは別に管理される場合がある。

### 1.7. Import Object and Reference Object

参照のみ可能なimport Objectと、子・属性を追加可能なreference objectとがある。

1. 参照のみ

   import/access

   - import したオブジェクトには、子や属性を追加することはできない。

2. 子・属性追加可能

   - 同一文書内で定義されたオブジェクトに対してのみ参照可能。前方参照も禁止しない。
   - クラス情報 ＋ `&` 指定により、そのIDは参照となる。
   - 子・属性を追加できる。

## 2. Gdoc Tag

Tagは、文書中の特定範囲に対して属性を付与し、文書中の情報を収集し、Gdoc Objectへ情報を保存する。

### 2.1. Block tag

テキストブロック要素に付与して、一つの Gdoc Object を生成する。
Block Tagは、Classとパラメータを指定することができる。

ex.

> #### 2.3. [@GSN:Goal] PROJECT GOALS
>
> ***[@import gdml.p.st3 from=gdml.md as=st3]***
>
> - ***[@Sys:Reqt er2]*** Application plugin system  \
>   Application should be addaptable.

#### 2.1.1. Format

Block Tagは、以下の形式で記述する。

```md
[@Class param1 param2 ...]
```

#### 2.1.2. Implicit Parameters

1. following text:
   タグ文字列の直後から改行あるいは次のタグまでを、following_textパラメーターとして扱う。
   オブジェクトタイプによって、簡潔な説明文や、名前として扱われる。

2. following lines:
   タグ文字列の次の行から、ブロック末尾あるいは次のタグまでを、following_linesパラメーターとして扱う。
   オブジェクトタイプによって、詳細な説明文や、名前として扱われる。

#### 2.1.3. Trailing Tag

ブロックタグの直前に２文字以上の連続する `-` がある場合、ブロック先頭からその`-`直前までのテキストをfollowing_linesパラメーターとして扱う。
この場合は後置きタグとなり、タグの後ろのfollowing linesは無視される。

   > Preceding line, preceding line.
   > Preceding text -- [@class id]

#### 2.1.4. Types

タグを付与場所・方法よって、生成されるオブジェクトの種類が異なる。

##### 2.1.4.1. Object Tag

セクション先頭以外のブロックに適用して、オブジェクトを生成する。

##### 2.1.4.2. Section Tag

セクション先頭のテキストブロックに適用して、オブジェクトセクションを構成する。

- ？ ブロックタグの付与された先頭ブロックに後続するブロックのうち、タグの付与されていないテキストブロックはセクションオブジェクトのtextプロパティとして扱われる。

##### 2.1.4.3. Import Tag

別の名前空間にあるオブジェクトへのショートカットを生成する。
import/accessに使用する。

##### 2.1.4.4. Reference Tag

同一文書内の別箇所にあるオブジェクトへの参照を生成する。

### 2.2. Inline tag

Inline文字列に付与して、対象要素をプロパティとしてセットする。

#### 2.2.1. Format

> ***@Note(1):***  \
> Some note to read.

#### 2.2.2. Implicit Parameters

*@definition(1):*  \
テキストブロック中にタグを付与すると、タグの次からブロック末まで、もしくは次のタグまでの文字列をプロパティとしてセットする。

1. following text:
   タグ文字列の直後から改行あるいは次のタグまでを、following_textパラメーターとして扱う。
   オブジェクトタイプによって、簡潔な説明文や、名前として扱われる。

2. following lines:
   タグ文字列の次の行から、ブロック末尾あるいは次のタグまでを、following_linesパラメーターとして扱う。
   オブジェクトタイプによって、詳細な説明文や、名前として扱われる。

#### 2.2.3. Trailing Tag

ブロックタグの直前に２文字以上の連続する `-` がある場合、ブロック先頭からその`-`直前までのテキストをfollowing_linesパラメーターとして扱う。
この場合は後置きタグとなり、タグの後ろのfollowing linesは無視される。

   > Preceding line, preceding line.
   > Preceding text -- @note(1):

### 2.3. Table tag

テーブル内のRange（セルの集合・範囲）をターゲットにしたタグ。
オブジェクト、プロパティを生成する。

#### 2.3.1. Range tag and Top tag

Rangeの左上角のセルに記述する。
テーブルの先頭セルにタグを記述すると、テーブル全体に影響する。

ex.

> | ***@Reqt*** | Name | Description |
> | ----- | ---- | ----------- |
> | er2 | Application plugin system | Application should be addaptable.

#### 2.3.2. Range And Top Cell

Range is two-dimensional array of cells in the table.

The biggest range in a table is the table itself.

Top cell is the topmost and leftmost cell.

## 4. PARSING RULE DETAILES

PandocAST形式の入力文書をパースするルールを定義する。

### 4.1. Input File Format

gdocは、PandocASTフォーマットのjsonデータを入力とする。

#### 4.1.1. Version

```
"pandoc-api-version": [1, 22]
```

- https://hackage.haskell.org/package/pandoc-types-1.22/docs/Text-Pandoc-Definition.html

- 現在 1.23。
  -> https://hackage.haskell.org/package/pandoc-types-1.23/docs/Text-Pandoc-Definition.html

- version 1.22 でない場合、gdocは警告を出力して処理を続行する。

#### 4.1.2. Generating PandocAST

- gfmをメインターゲットとする。

- `pandoc -f gfm+sourcepos -t html | pandoc -f html -t json`

  `+sourcepos` オプションによる、ソースマッピング情報に対応する。

  一旦htmlを経由することで、`<br>` をLineBreakとして解釈するなど埋め込まれたhtmlによる記述をPandocAst要素として取り込む。

  - 現在 `+sourcepos` に対応しているのは gfm と commonmark のみであり、commonmark はテーブルに対応しない。
  - 他のタイプでも動作するが、完全なテスト（特にgfmに存在しない要素タイプのテスト）は行っていない。

- [ ] +sourcepos による振る舞いの違い（todoリストなど）、どこかに記述できる場所をつくる。

### 4.2. PandocAST From GDML's View

#### 4.2.1. PandocAST Element Types

Pandoc AST は、以下のような要素タイプを持ち、DOMに近いものになっている。

1. Block Element Types

   PandocASTにおいて、文書全体はブロックのリストである。

   ブロックは種別をもち、そのブロック自身がブロックのリストを構成するものもある。

   Para, plane, Table, OrderedList, BulletList, Headerなど。

2. Inline Element Types

   Paragraph, Planeのコンテンツは、Inline要素のリストである。

   Inline要素は種別をもち、Inlineのリストを構成するものがある。
   デコレーターも、このインライン要素のリストを構成するインライン要素である。

   Str, Space, LineBreak, Strong, Emphasis, Codeなど。

#### 4.2.2. Document Structure

Pandoc AST is a great data structure for format conversion, but it is a bare data structure that is not necessarily intuitive for human reading and writing.

Therefore, GDML covers Pandoc AST with a thin abstract concept and considers a document as the following structure.

This is called the Gdoc data structure, and a document that follows this structure is called a Gdoc.

GDML is a markup language for Gdoc documents.

<br>

| [![](./_puml_/GdocMarkupLanguage/GdocObjectStructure.png)](./GdocMarkupLanguage.puml) |
| :-----: |
| [@fig 1.1] Gdoc Data Structure [Overview] |

<br>

Gdoc Markup Notation では便宜上、Header, Para, Plane, LineBlock の４タイプを、テキストブロックと呼ぶ。

- 装飾が無視されること、Strikeout は無視されること、などを例示する。

##### 4.2.2.1. Section

Headerで区切られ階層化した論理的なブロックリストを Sectionと呼ぶ。

Pandoc ASTのデータ構造に Section の概念はない。
ブロックリストは階層化しない一次配列であり、Headerブロックもまたその要素の１つである。

gdocはHeaderブロックをセクションの開始とみなして、階層化したブロックリスト構造として扱う。

##### 4.2.2.2. ListBlock

ListBlockには、Pandoc ASTのBulletListとOrderedListとがある。
リストの各項目は、それぞれブロックのリストで構成される。

gdocは、ListBlockに対するタグ（キャプションタグ）が付与されていない限り単に階層化したBlockListとみなし、解析中にはこの階層を再帰的に解析対象とみなす。

つまり、Section も ListBlock も、Gdoc にとっては階層化したBlockListである。

*@eng:*  \
In other words, "Sections" and "ListBlock" are both hierarchical BlockLists for Gdoc.

##### 4.2.2.3. Object Section

Sectionあるいはリストアイテムのうち、先頭ブロックにタグを付与されたものは GdObject が生成され、名前空間を提供するなどオブジェクトのコンテクストとして機能する。

Gdoc では、このコンテクスト空間を Object Section とよび、空間を構成する GdObject を Section Object と呼ぶ。

- [ ] 以下、適切な場所へ移動する

> Object Section は、以下の機能をもつ。
>
> 1. 名前空間の提供
>
>    文書中に付与されたtagによりGdObjectが生成されるとき、そのタグが所属する Object Section が親となる。
>    生成されるオブジェクトの id は、この親オブジェクトのなかで一意になる。
>
> 2. デフォルトカテゴリの指定
>
>    tag で指定する Class について、デフォルトのカテゴリ情報を保持する。
>    Class にタイプだけが指定された場合、以下のように特定される。
>
>    1. Object セクションが保持するデフォルトカテゴリから探す。
>    2. 見つからない場合、上位のObject Sectionが保持するデフォルトカテゴリから探す。
>    3. 繰り返し



### 4.3. Retrieving Blocks

PandocASTの文書全体は、連続するブロックのリストである。
この時点ではヘッダーによるセクション分割・階層化は行われておらず、ヘッダーは単なる１つのブロックである。
<small>*@trace.inContextOf:* gdon.p.h.s</small>

Gdocはこのリスト内のブロックを先頭から順に取り出し、解析する。
そのブロック取り出しの際、以下のルールに従う。

<br>

| [![](./_puml_/GdocMarkupLanguage/SectionDataStructure.png)](./GdocMarkupLanguage.puml) |
| :-----: |
| [@fig 1.1] Section Data Structure |

<br>

#### 4.3.1. Div Block

- BlockListから取り出したBlockがDiv Blockであった場合、これによる階層構造は無視し、Div Blockの子要素をDiv自身の位置に展開する。
- Divが階層化している場合も、再帰的にこれを展開する。

#### 4.3.2. BulletList/OrderedList

BulletList/OrderedListの各リストアイテムは、文書全体と同様にブロックのリストである。

##### 4.4.3.2. Detecting tags

- Block tag

  RE = `\[@.*?(".*?(\\".*?)*?".*?)*?\]`

  ```py
  r"""
  \[@             # 1. Tag starts with '[@'.
    .*?           #    2. Tag may include chars.
    (             #    3. Tag may include Quoted strings and following chars.
      \"          #       4. Quoted str starts with '"'.
        .*?       #          5. Quoted str may include chars.
        (         #
          \\\"    #          6. Quoted str may include escaped '"'s
          .*?     #             and following chars.
        )*?       #
      \"          #       7. Quoted str ends with '"'
      .*?         #          and following chars.
    )*?           #
  \]`             # 8. Tag ends with ']'
  """
  ```

  - L_BLOCK_TAG = `r"\[@"`
  - R_BLOCK_TAG = `r"\]"`

- Inline tag

  RE = `(?:^|\s)@(\w|#)*?(\(.*?(".*?(\\".*?)*?".*?)*?\))?:(?=\s|$)`

  ```py
  r"""
  (?:^|\s)@         # 1. Tag starts with (^ or space) and '@'.
    (\w|[#])*?      #    2. Tag name may follow tag header without space char.
    (\(.*?          #    3. Tag may have args in '()'.
      (             #    4. Args may include Quoted strings and following chars.
        \"          #       5. Quoted str starts with '"'.
          .*?       #          6. Quoted str may include chars.
          (         #
            \\\"    #          7. Quoted str may include escaped '"'s
            .*?     #             and following chars.
          )*?       #
        \"          #       8. Quoted str ends with '"'
        .*?         #          and following chars.
      )*?           #
    \))?            #    9. Tag name(or Args) trail NO chars.
  :(?=\s|$)         # 10. Tag ends with ':' and have no trailing chars.
  """
  ```

  - L_INLINE_TAG = `r"(^|\s)@"`
  - R_INLINE_TAG = `r":(?=\s|$)"`

[@]

- @tag(key=" \" "):  @tag( "" ): id id
